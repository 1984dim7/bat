# バット工房 iOSアプリ版 — 開発指示書 & 計画

このドキュメントは、現行の `bat`（単一HTML / Three.js r128 / Capacitorスキャフォルド済み）を
**別プロダクト**として複製し、グラフィックを全面刷新したうえでApp Storeアプリとして
リリースするための指示書である。実装担当（人間 or AIエージェント）はこれを上から順に実行する。

---

## 0. 前提と現状サマリ

| 項目 | 現状 |
|---|---|
| コード | `index.html` 単一ファイル 約2,620行 |
| 描画 | Three.js r128（CDN→`vendor/`にローカル化済み） |
| ジオメトリ | 全て手続き生成（バット=LatheGeometry、観客=Points、球場=Box/Plane） |
| マテリアル | 大半が `MeshLambertMaterial` / 頂点カラー（PBRではない） |
| ポストエフェクト | なし |
| 物理 | 自前実装（質量/重心/慣性/COP/しなり/反発）— **流用必須・再実装禁止** |
| iOS | Capacitor 8（SPM方式）でスキャフォルド済み、未ビルド |
| ランキング | GAS + スプレッドシート共有（稼働中） |

**死守ライン**: ゲームバランス（物理計算・飛距離の指数関数・スイング補正・球速カーブ）は
現行の数値を1つも変えない。グラフィック刷新はあくまで「見た目」レイヤーに閉じる。
物理に手を入れると `refactor-instructions.md` の「Behaviors To Preserve」が全部壊れる。

## 1. リポジトリ複製（Day 0）

新プロダクトなので**ブランチではなく新リポジトリ**にする。履歴は引き継いでよい。

```bash
# 1. 既存リポジトリを複製
cd ~/Desktop/Coding
cp -R Bat BatCraftApp
cd BatCraftApp

# 2. gitリモートを張り替え（GitHubで空のbat-app等を先に作っておく）
git remote remove origin
git remote add origin https://github.com/1984dim7/bat-app.git

# 3. アプリ識別子を変える（Web版と衝突させない）
#    capacitor.config.json の appId / appName
#    例: jp.dim7.batcraft → jp.dim7.batcraft.app

# 4. 初回コミット
git add -A && git commit -m "Fork from bat: iOS app edition"
git push -u origin main
```

- Web版（dim7.jp / `bat`リポジトリ）は**そのまま残す**。アプリ版は別系統として育てる。
- 共有ランキングは**同じGAS APIを使い回す**（Web版とアプリ版で同一ランキングに乗る）。
  分けたい場合はGASを複製して別URLにする。判断は §6 で。

## 2. アーキテクチャ決定（最重要・Day 0で確定）

グラフィック刷新の野心度で2択になる。**先にこれを決めないと後の全作業が変わる。**

### 案A: Capacitor継続 + Three.js刷新（推奨）
- WKWebView上でThree.jsを動かす現構成のまま、描画だけ強化する。
- **利点**: 既存コード資産（物理・UI・ランキング）を100%流用。Web版と同一コードベースで両方リリースできる。工数最小。
- **欠点**: WebGLの上限性能に縛られる（とはいえこのゲームは軽いので実害ほぼなし）。
- **適性**: 「今のゲームを綺麗にしてアプリにしたい」なら最適。

### 案B: SceneKit/RealityKitでネイティブ再実装
- 描画・シーンをSwiftで書き直し、物理ロジックだけJSから移植 or 再計算する。
- **利点**: Metal直叩きで最高品質の描画・120fps・ハプティクスとの密結合。
- **欠点**: 工数が一桁大きい（数ヶ月）。物理の移植でバランスが壊れるリスク。Web版と乖離する。
- **適性**: 「フォトリアルな野球ゲームにして本気で売る」なら検討。今回はオーバースペック。

> **推奨は案A。** 以降の計画は案A前提で書く。
> Three.jsはWebGLでもPBR・Bloom・影・ポストエフェクトが十分使え、
> 「グラフィック刷新」の要求は案Aで満たせる。案Bは将来の続編で。

## 3. グラフィック刷新計画（案A / 本計画の中核）

現状の「手続き生成＋Lambert」を、**PBR＋ポストプロセス＋テクスチャ**に引き上げる。
作業は描画レイヤーに閉じ、ジオメトリの当たり判定座標やバットのLathe形状は変えない。

### 3-0. 基盤: Three.jsバージョン方針
- **r128のまま据え置く**ことを第一候補とする。最新版(r160+)はAPI破壊変更が多く、
  2,600行が全面修正になる。r128でもPBR/Bloom/影は揃っている。
- どうしても新機能（WebGPURenderer等）が要るなら別タスクで段階移行。本計画では扱わない。

### 3-1. マテリアルのPBR化（効果大・リスク小）
- `MeshLambertMaterial` → `MeshStandardMaterial`（metalness / roughness）へ置換。
- バット素材ごとに質感を作り分ける（現行は色だけ違う）:
  | 素材 | metalness | roughness | 追加 |
  |---|---|---|---|
  | 木製 | 0.0 | 0.7 | 木目テクスチャ |
  | カーボン | 0.1 | 0.3 | 綾織りノーマルマップ |
  | アルミ | 0.9 | 0.25 | 環境マップ反射 |
  | 金 | 1.0 | 0.15 | 金色env、強反射 |
  | ダイヤ | 0.2 | 0.0 | 透過 + envMap |
  | 黒曜石 | 0.6 | 0.05 | 黒く磨いた反射 |
- **環境マップ(envMap)** を1枚入れるだけで金属・ガラス系が劇的に良くなる。
  ナイター球場のHDRIまたは簡易キューブマップを用意。

### 3-2. ポストプロセス（夜のスタジアムが映える）
- `EffectComposer` + `UnrealBloomPass` を導入。
  - 照明塔・スコアボード・打球のグロー・月・星が発光して一気に「映え」る。
  - 既に発光体（ball halo, ライト）があるので相性抜群。
- `SMAAPass` か MSAA でジャギ低減（モバイル解像度で効く）。
- 注意: ポストプロセスはfill-rate負荷が高い。iPhone実機で60fps維持を必ず確認。
  解像度スケールを `renderer.setPixelRatio(Math.min(devicePixelRatio, 2))` で抑える。

### 3-3. テクスチャ & ディテール
- 芝: 現状ベタ塗り → 芝目テクスチャ + 内野/外野の刈り分けストライプ。
- 土（マウンド・ベース周り）: ラフネステクスチャ。
- スコアボード: 現状Canvas動的描画を維持しつつ発光させる。
- 観客: 現状 `Points` → 簡易インスタンスメッシュ（板ポリ）にすると密度感が出る（任意・重め）。

### 3-4. ライティング
- 現状の単純ライト → ライト塔4基からの指向性ライト + 弱い環境光。
- シャドウマップを主要オブジェクト（バット・ボール・打者）に限定して有効化。
  全面シャドウはモバイルで重いので**選択的に**。

### 3-5. エフェクト（打撃の気持ちよさ）
- ジャストミート時: パーティクル火花 + 一瞬のフラッシュ + 軽いカメラシェイク。
- 場外/宇宙到達: 既存演出を強化（リング状の衝撃波スプライト等）。
- 打球トレイル: 現状の球列 → ラインジオメトリ + Bloomで光の軌跡に。

### 3-6. UIリニューアル
- 現行UIはHTML/CSSオーバーレイ。アプリ品質に合わせて:
  - フォント・余白・角丸・グラスモーフィズムを統一トーンに。
  - セーフエリア対応は実装済み。アイコンをSF Symbols風 or オリジナルに差し替え。
  - クラフト/打席/結果のトランジション演出を追加。

> **刷新の優先順位（費用対効果順）**: 3-2 Bloom → 3-1 PBR+envMap → 3-5 打撃エフェクト
> → 3-3 テクスチャ → 3-4 影 → 3-6 UI。上3つだけで体感は別物になる。

## 4. iOSアプリ化の作業（Capacitor / 案A）

スキャフォルドは済んでいる。ここはネイティブ品質の詰め。

### 4-1. ビルド前提（人間作業）
- **Xcode** をApp Storeからインストール（未導入。約10GB）。
- `sudo xcode-select -s /Applications/Xcode.app`
- Apple Developer Program 登録（年99 USD、ストア配布に必須）。

### 4-2. ネイティブ機能（アプリならではの価値 = 審査対策も兼ねる）
1. **ハプティクス** `@capacitor/haptics`: ジャストミートで `impact(Heavy)`、ヒットで `Medium`。
   打撃ゲームと最高相性。これだけでアプリ化の意義が立つ。
2. **消音スイッチ対策**: WebAudioはサイレントモードで鳴らない。
   `AVAudioSession` を `.playback` に設定するネイティブ初期化を `AppDelegate.swift` に追加。
3. **Game Center**（任意）: 共有ランキングをGASからGame Centerリーダーボードへ昇格。
   「本物の公開ランキング」になる。GASは残してハイブリッドでもよい。
4. **起動画面・アイコン**: LaunchScreenとAppIconを新グラフィックのトーンで作成。

### 4-3. パフォーマンス確認
- ProMotion(120Hz)実機で物理が破綻しないか（dtベースなので原理上OK、要実測）。
- Bloom有効時のfill-rate。発熱・フレーム落ちを iPhone 実機でプロファイル。
- 低スペック端末向けに「描画品質 高/低」トグルを用意（Bloom/影をオフにできる）。

### 4-4. ワンコマンド運用（整備済み）
```bash
npm run ios   # build → cap sync → Xcode起動
```

## 6. 確定が必要な判断事項（実装前に人間が決める）

1. **アーキテクチャ**: 案A（Capacitor継続・推奨）か案B（ネイティブ）か。→ 推奨A。
2. **ランキング**: Web版と共有（同一GAS）か、アプリ専用に分離（GAS複製）か。
3. **グラフィック方向性**: フォトリアル寄りか、現行のネオン・ナイター様式を強化する方向か。
   → 既存資産を活かすなら後者（ネオン強化）が低コストで映える。
4. **マネタイズ**: 完全無料か、買い切りか、広告か。審査要件と実装が変わる。
5. **対象端末**: iPhoneのみか iPad対応もか（現UIは縦固定。iPad横は別レイアウトが要る）。

## 7. やらないこと（スコープ外）

- 物理エンジンの差し替え・ゲームバランスの変更（死守ライン）。
- Three.js最新版への全面移行（別タスク）。
- Android版（Capacitorなので将来 `cap add android` で展開可能だが今回は対象外）。
- オンライン対戦・サーバー権威化（ランキングは引き続きクライアント申告 + GAS側バリデーション）。
